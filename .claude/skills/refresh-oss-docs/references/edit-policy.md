# Edit policy

How aggressively an editor may change a page depends on how much *editorial*
content that page carries. A mechanically-correct edit can silently destroy a
tutorial while being exactly right for a reference page.

## The four types

| Type            | Substrate         | Policy           |
| --------------- | ----------------- | ---------------- |
| **reference**   | hand-written .mdx | **Assertive**    |
| **how-to**      | .ipynb            | **Assertive**    |
| **tutorial**    | .ipynb            | **Conservative** |
| **explanation** | hand-written .md  | **Conservative** |

### Assertive

The snapshot signature is ground truth. Rewrite entries, update call sites,
adjust the surrounding steps so the page stays coherent and runnable. Add
reference entries for new symbols.

### Conservative

Make only the edits the API change *forces*:

- a renamed symbol or parameter → rename it
- a reordered argument → reorder it
- a changed default asserted in prose → correct the value

Nothing else. Do not restructure, do not rewrite narrative, do not "improve"
prose you happen to be passing through.

## Escalation

**Escalation is not failure.** A refresh that updates 8 pages and flags 2 is a
good outcome. One that silently rewrites all 10 is not.

Flag — do not edit — when:

| Type        | Flag when                                                    |
| ----------- | ------------------------------------------------------------ |
| reference   | never (the work is mechanical)                               |
| how-to      | the delta removes the feature the guide exists for           |
| tutorial    | the change alters *what the lesson teaches*                  |
| explanation | the concept being explained no longer exists                 |

A `symbol_removed` delta on a page is the most common escalation trigger: the
page is now *wrong*, not merely stale, and deciding what should replace it is an
editorial call, not a mechanical one.

## New symbols

Add a reference entry — that is the reference contract.

Do **not** author tutorials, how-tos, or explanations for new features. Whether a
feature deserves a page, and what that page should teach, is a human's decision.
New symbols with no existing mention are reported in the PR body as
"undocumented", and there they stop.

## Notebooks

If your work order names a `.ipynb`, edit the notebook. The `.mdx` beside it is
git-ignored build output regenerated by `make regen-mdx`; an edit there
evaporates.

Edit the `source` array of the cell the work order points at. Leave `outputs`
and `execution_count` alone — the verifier re-executes notebooks, and CI
refreshes outputs on merge.

## Both substrates

- Preserve frontmatter, custom prose, and hand-written examples.
- Match the surrounding style: comment density, naming, and idiom.
- Do not reformat lines you did not need to change.
